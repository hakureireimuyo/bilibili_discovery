# 网页融合第一阶段开发任务清单

本文档用于把 [`网页融合核心模块规范`](/D:/ProjectFloder/TS/bilibili_discovery/docs/网页融合核心模块规范.md) 拆解成可直接执行的开发任务。

本阶段默认目标：

- 先建立一个独立的网页融合子系统
- 与当前 `src/content` 采集链路保持平行
- 优先打通五个核心能力的最小闭环
- 先轻依赖数据层，后续再逐步深化

本清单聚焦第一阶段，不覆盖：

- 深度个性化排序
- 复杂系列识别
- 页面曝光 / 反馈事件全量入库
- 自定义样式生态

---

## 1. 任务总览

建议按 8 个阶段推进：

1. 目录与运行时骨架
2. 页面识别与生命周期
3. 配置与预设模式
4. 页面数据提取与卡片抽象
5. 卡片评估与推荐流过滤
6. 卡片状态标记与悬浮信息卡片
7. 视频页极简模式
8. 数据层轻支撑与验收

每个阶段完成后都应保证：

- TypeScript 可编译
- 不破坏当前内容采集逻辑
- 页面增强功能可单独关闭
- 页面结构识别失败时可安全降级

---

## 2. 阶段一：目录与运行时骨架

这一阶段的目标不是实现功能，而是先把网页融合子系统的边界搭起来。

### Task 1.1 新建网页融合目录

目标：

- 新建 `src/web-integration/`

建议子目录：

- `runtime/`
- `extractors/`
- `evaluators/`
- `overlays/`
- `filters/`
- `focus-mode/`
- `presets/`
- `services/`
- `types/`

完成标准：

- 目录结构清晰
- 命名与规范文档一致
- 不与 `src/ui`、`src/content` 职责混淆

### Task 1.2 新建网页融合入口文件

目标：

- 新建网页融合运行时入口

建议文件：

- `src/web-integration/index.ts`

需要完成：

- 定义统一初始化入口
- 暴露 `start()` / `stop()` 或等价生命周期方法
- 预留子模块挂载位置

完成标准：

- 子系统可以独立初始化
- 即使没有任何具体功能，也能安全运行和退出

### Task 1.3 在 content script 中接入网页融合入口

目标文件：

- `src/content/index.ts`

需要完成：

- 在现有追踪器初始化之外，启动网页融合子系统
- 保持与采集系统平行，而不是塞进现有 trigger/collector 链路

完成标准：

- 现有视频追踪、关注、收藏、互动采集不受影响
- 网页融合初始化失败不会阻断现有 content script

### Task 1.4 定义网页融合基础类型

目标：

- 新建 `src/web-integration/types/`

建议优先定义：

- 页面类型
- 运行时上下文
- 卡片上下文
- 模块配置快照
- 卡片评估结果

完成标准：

- 网页融合系统内部不依赖大量 `any`
- 类型命名和现有项目风格一致

## 3. 阶段二：页面识别与生命周期

这一阶段负责解决“系统在哪个页面工作、页面变化时如何重挂载”的问题。

### Task 2.1 实现页面识别模块

目标文件建议：

- `src/web-integration/runtime/page-detector.ts`

需要完成：

- 识别首页
- 识别搜索页
- 识别分区页
- 识别视频页
- 识别 UP 页
- 提供统一页面类型枚举

完成标准：

- 在典型 B 站页面下能稳定返回页面类型
- 未识别页面时返回安全兜底类型

### Task 2.2 实现 URL 变化监听

目标文件建议：

- `src/web-integration/runtime/navigation-watcher.ts`

需要完成：

- 监听 SPA 场景下的 URL 切换
- 在页面类型变化时通知网页融合运行时

完成标准：

- 同页面内路由切换后，系统能重新判断页面上下文
- 不依赖整页刷新

### Task 2.3 实现页面生命周期管理器

目标文件建议：

- `src/web-integration/runtime/integration-runtime.ts`

需要完成：

- 管理模块挂载
- 管理模块卸载
- 管理重新初始化
- 统一注册和清理 `MutationObserver`

完成标准：

- 页面切换时不会重复挂多个同类模块
- 模块卸载后不会残留大量事件监听

### Task 2.4 实现失败降级机制

目标文件建议：

- `src/web-integration/runtime/error-boundary.ts`

需要完成：

- 某个模块初始化失败时只关闭该模块
- 不让整个网页融合系统崩掉
- 不影响原有内容采集系统

完成标准：

- 单模块故障可隔离
- 日志中能看出失败来源

## 4. 阶段三：配置与预设模式

这一阶段负责给整个子系统建立统一的配置入口和模式切换能力。

### Task 3.1 定义网页融合设置模型

目标文件建议：

- `src/web-integration/types/settings.ts`
- `src/web-integration/presets/preset-types.ts`

需要定义：

- 网页融合总开关
- 推荐流过滤设置
- 卡片状态标记设置
- 悬浮卡片设置
- 视频页极简设置
- 当前预设模式

完成标准：

- 字段结构清晰
- 支持后续扩展
- 不与现有基础应用设置字段混杂

### Task 3.2 实现配置读取服务

目标文件建议：

- `src/web-integration/services/integration-settings-service.ts`

需要完成：

- 从当前 settings / `APP_META` 读取网页融合配置
- 提供默认配置
- 提供配置规范化方法

完成标准：

- 即使用户没有配置过，也能拿到完整默认值
- 非法配置会自动回退

### Task 3.3 实现预设模式定义

目标文件建议：

- `src/web-integration/presets/preset-definitions.ts`

建议内置：

- 高质量模式
- 长视频优先模式
- 学习模式
- 纯净模式

需要完成：

- 每个模式定义自己的过滤规则
- 每个模式定义自己的标记策略
- 每个模式定义自己的视频页极简强度

完成标准：

- 模式不只是名字，而是完整规则包
- 模式之间差异明确

### Task 3.4 实现模式解析服务

目标文件建议：

- `src/web-integration/presets/preset-resolver.ts`

需要完成：

- 将“当前模式 + 用户配置”解析为最终配置快照
- 支持模式级覆盖默认设置

完成标准：

- 子模块只消费“解析后快照”
- 不需要各自理解模式规则

### Task 3.5 实现模式变更广播

目标文件建议：

- `src/web-integration/runtime/runtime-events.ts`

需要完成：

- 模式切换后通知各子模块
- 页面内所有模块响应同一份最新配置

完成标准：

- 模式切换后过滤、标记、极简模式都能同步更新
- 不需要刷新页面

## 5. 阶段四：页面数据提取与卡片抽象

目标文件建议：

- `src/web-integration/types/card-context.ts`

需要定义：

- 卡片 DOM 引用
- BV 或其他唯一标识
- 标题
- UP 信息
- 时长
- 封面
- 来源类型
- 标签入口信息

完成标准：

- 结构足够支撑过滤、标记、悬浮卡片三类模块

### Task 4.2 实现推荐流卡片提取器

目标文件建议：

- `src/web-integration/extractors/card-extractor.ts`

需要完成：

- 在首页识别卡片
- 在搜索页识别卡片
- 在分区页识别卡片
- 提取卡片基础信息

完成标准：

- 能批量返回当前页面卡片列表
- 页面局部刷新后能重新识别

### Task 4.3 实现视频页区块提取器

目标文件建议：

- `src/web-integration/extractors/video-page-extractor.ts`

需要完成：

- 识别评论区
- 识别相关推荐区
- 识别互动数据区
- 识别主要信息区块

完成标准：

- 视频页极简模式所需区块都能被抽象访问

### Task 4.4 实现卡片数据补全服务

目标文件建议：

- `src/web-integration/services/card-data-service.ts`

需要完成：

- 对基础卡片上下文进行补全
- 必要时调用已有视频 API 获取详情
- 做简单缓存，避免重复请求

完成标准：

- 卡片 hover 时能获得更完整信息
- 不会因为频繁鼠标移动造成请求风暴

### Task 4.5 实现页面卡片监听

目标文件建议：

- `src/web-integration/runtime/card-list-observer.ts`

需要完成：

- 监听页面卡片列表新增 / 替换
- 通知后续评估和渲染模块重新处理

完成标准：

- 滚动加载新卡片后系统仍然有效
- 不需要用户手动刷新

需要验证：

- 首页识别正确

需要验证：

- 评论区可隐藏

## 11. 推荐的执行顺序

如果按最稳妥的实际研发顺序，建议这样推进：

1. 阶段一：目录与运行时骨架
2. 阶段二：页面识别与生命周期
3. 阶段三：配置与预设模式
4. 阶段四：页面数据提取与卡片抽象
5. 阶段五：卡片评估与推荐流过滤
6. 阶段七：视频页极简模式
7. 阶段六：卡片状态标记与悬浮信息卡片
8. 阶段八：数据层轻支撑改造
9. 阶段九：联调与验收

这样排序的原因是：

- 先把运行时和页面识别打稳
- 再把过滤和极简模式这类最直接的功能先打通
- 最后再叠加状态标记和悬浮信息卡片这类解释层能力

---

## 12. 最小可交付版本

如果希望尽快做出网页融合第一版，建议最小闭环是：

1. 建立 `src/web-integration/` 目录和运行时入口
2. 能识别首页 / 视频页
3. 能读取当前预设模式
4. 能识别推荐卡片
5. 能执行基础推荐流过滤
6. 能在视频页启用基础极简模式

这个最小版本先不要求：

- 完整悬浮信息卡片
- 基于历史行为的已看过判断
- 复杂缓存
- 数据层新增专门 store

只要这六步打通，网页融合第一阶段就已经成立。

---

## 13. 第二个可交付版本

在最小版本稳定后，建议补齐第二个闭环：

1. 卡片评估结果统一输出
2. 卡片状态标记上线
3. 悬浮信息卡片上线
4. 模式切换可以动态影响全部模块

做到这一步后，插件就不只是“页面能被改”，而是开始具备“可解释的浏览辅助”能力。

---

## 14. 一句话结论

网页融合第一阶段的正确拆法，不是直接同时做五个大功能，而是先做“运行时底座 + 页面识别 + 配置模式 + 卡片抽象”，再把过滤、极简、标记和悬浮解释逐步挂上去。

这样开发顺序最稳，返工也最少。
