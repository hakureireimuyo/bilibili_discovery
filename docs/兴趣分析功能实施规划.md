# 兴趣分析功能实施规划

本文档用于指导后续 AI 智能体在当前项目中实现“兴趣分析”功能。

本版规划采用更干净的方案：

- `Category` 不参与兴趣分析
- 不复用当前 `WatchEvent` 作为兴趣分析主模型
- 单独建立“兴趣贡献事件流”
- 视频只作为归一化时的中间信息，可选保留追溯字段，但不是兴趣统计核心对象

这套方案的目标是让兴趣分析拥有独立、稳定、可解释、便于聚合的长期数据结构。

---

## 1. 当前项目现状

### 1.1 当前项目结构

当前项目的相关目录职责已经比较清晰：

- `src/database/types`

- `src/database/indexeddb/config.ts`

- `src/database/implementations`
  - 负责数据库原子读写与聚合实现
- `src/content/processors/data-processor.ts`
  - 当前采集数据入库主链路
- `src/engine/llm-client.ts`
  - 现有 LLM 调用入口
- `src/ui/watch-stats`
  - 当前观看统计页面

### 1.2 当前已有可复用数据

兴趣分析虽然不直接复用 `WatchEvent` 作为主存储，但仍会使用当前这些数据做“归一化来源”：

- `Video`
  - 含 `title / description / tags / creatorId / duration`
- `Tag`
  - 当前系统统一标签库
- `FavoriteVideoEntry`
  - 可作为兴趣加强信号
- `Creator`
  - 可作为辅助语义来源
- `WatchEvent`
  - 当前仅作为输入来源之一，不作为最终兴趣分析事实表

### 1.3 当前必须遵守的约束

- `Category` 只服务标签过滤，不参与兴趣分析
- 不应在 UI 页面直接实现复杂数据聚合
- 不应让 LLM 进入实时主链路
- 兴趣分析要尽量独立于当前 `WatchEvent` 的实现缺陷

## 2. 为什么采用“兴趣贡献事件流”方案

你当前项目如果直接围绕 `WatchEvent` 做兴趣分析，会遇到两个问题：

- 当前 `WatchEvent` 实现更像“按视频累计更新”，不是逐次观看事件
- 兴趣分析真正关心的是“用户把时间投入到了哪些语义主题”，而不是“具体看了哪个 BV”

因此更合理的方式是：

1. 原始层继续采集视频、tag、观看数据
2. 归一化层把这些原始行为转换成“兴趣贡献事件”
3. 统计层只基于“兴趣贡献事件”做窗口聚合

这带来的收益：

- 兴趣分析与原始行为存储解耦
- 时间窗口统计更简单
- 后续支持收藏、点赞、搜索等更多兴趣信号更自然
- 兴趣画像的长期结构更清晰

---

## 3. 功能目标

第一阶段目标：

- 基于近期观看和收藏行为生成用户兴趣画像
- 用有限的大类兴趣替代过细的原始 tag
- 支持 `7d` 和 `30d` 两个时间窗口
- 支持兴趣占比、兴趣强度、兴趣变化趋势
- 支持解释兴趣来自哪些 tag、哪些行为

第一阶段不做：

- 不做完整推荐系统
- 不做复杂图谱
- 不做自由生成主题体系
- 不依赖逐次完整观看日志

---

## 4. 总体设计

兴趣分析采用四层结构：

### 4.1 兴趣主题层

定义有限且稳定的兴趣大类。

例如：

- 游戏
- 动画
- 音乐
- 科技
- 数码
- 知识
- 教程
- 生活
- 影视
- 二次元
- 搞笑
- 其他

### 4.2 语义映射层

把当前系统里的原始 `Tag` 映射到兴趣主题。

例如：

- `鬼泣5 -> 游戏`
- `VOCALOID -> 音乐`
- `MMD -> 二次元 + 音乐`

### 4.3 兴趣贡献事件层

这是本方案最核心的一层。

每条记录表达的是：

> 某个时间点，某种行为，对某个兴趣主题贡献了多少观看时长或兴趣权重

这层不再以视频为主键思维，而以“兴趣贡献”为中心。

### 4.4 兴趣快照层

从兴趣贡献事件中汇总出：

- 近 7 天兴趣画像
- 近 30 天兴趣画像
- 上升/下降趋势

## 5. 新增数据模型规划

建议新增：

- `src/database/types/interest.ts`

### 5.1 `InterestTopic`

```ts
export interface InterestTopic {
  topicId: ID;
  name: string;
  description?: string;
  parentTopicId?: ID;
  isActive: number;
  createdAt: Timestamp;
  updatedAt: Timestamp;
}
```

用途：

- 定义兴趣大类本体
- 允许后续做层级，但第一阶段以一级大类为主

### 5.2 `TagInterestMapping`

```ts
export interface TagInterestMapping {
  mappingId: ID;
  tagId: ID;
  topicId: ID;
  score: number;
  source: 'rule' | 'llm' | 'manual';
  confidence: number;
  version: number;
  createdAt: Timestamp;
  updatedAt: Timestamp;
}
```

用途：

- 将细粒度 tag 映射到兴趣主题
- 支持一对多映射

说明：

- `score` 表示 tag 对某 topic 的归属强度
- `confidence` 表示当前映射结论的置信度

### 5.3 `InterestContributionEvent`

```ts
export interface InterestContributionEvent {
  contributionEventId: ID;
  platform: Platform;
  topicId: ID;
  sourceType: 'watch' | 'favorite' | 'like' | 'coin' | 'manual';
  eventTime: Timestamp;
  dateKey: string;

  watchDuration: number;
  contributionScore: number;

  progress?: number;
  isComplete?: number;

  tagIds: ID[];
  tagNames?: string[];

  videoId?: ID;
  creatorId?: ID;

  createdAt: Timestamp;
}
```

这是第一阶段最核心的新类型。

字段说明：

- `topicId`
  - 指向最终兴趣主题
- `sourceType`
  - 表示该贡献来自观看、收藏、点赞等何种行为
- `watchDuration`
  - 该主题实际分摊到的观看时长
- `contributionScore`
  - 经过权重计算后的最终贡献值
- `tagIds`
  - 保留该贡献的原始 tag 来源
- `videoId`
  - 可选，仅用于调试追溯，不作为主统计依赖
- `creatorId`
  - 可选，用于未来做主题下的创作者偏好分析

### 5.4 `InterestSnapshot`

```ts
export interface InterestSnapshot {
  snapshotId: ID;
  platform: Platform;
  dateKey: string;
  window: '7d' | '30d';
  topicId: ID;
  watchDurationScore: number;
  eventCountScore: number;
  favoriteScore: number;
  finalScore: number;
  ratio: number;
  sourceEventCount: number;
  updatedAt: Timestamp;
}
```

用途：

- 缓存窗口聚合结果
- 供 UI 快速消费

### 5.5 可选：趋势结果类型

第一阶段可不落库存储，只在实现层动态计算。

如果后续需要长期趋势序列，再新增独立类型。

## 8. 语义映射策略

### 8.1 核心思路

映射链路是：

`Tag -> InterestTopic -> InterestContributionEvent`

也就是说：

- 视频只是生成贡献事件时的原始信息来源
- 真正入兴趣分析事实层的是“贡献事件”

### 8.2 Tag 映射流程

处理顺序：

1. 先查 `TagInterestMapping`
2. 若命中，直接复用
3. 若未命中，先走规则映射
4. 规则失败时，再走 LLM 映射
5. 将结果持久化到 `TagInterestMapping`

### 8.3 规则映射优先

第一阶段建议优先积累一份规则映射表。

优先覆盖方向：

- 游戏
- 音乐
- 科技/数码
- 知识/教程
- 动画/二次元
- 搞笑/鬼畜

### 8.4 LLM 仅用于补洞

LLM 只在这些情况触发：

- tag 不在规则映射表中
- tag 语义过于模糊
- 视频标题/简介对 tag 有明显补充价值

LLM 输出必须强制约束为固定主题集合内的 JSON 结果。

## 9. 兴趣贡献事件生成规则

### 9.1 总体原则

兴趣分析不直接用“视频”做统计单位，而是把一次行为拆成一个或多个兴趣贡献事件。

例如某视频映射结果为：

- 游戏 `0.8`
- 二次元 `0.2`

如果这次用户观看了 600 秒，则可生成两条贡献事件：

- 游戏：`watchDuration = 480`
- 二次元：`watchDuration = 120`

### 9.2 贡献分数计算

建议公式：

`contributionScore = baseScore * completionFactor * sourceFactor * recencyFactor`

其中：

- `baseScore`
  - 默认由分摊后的观看时长得出
- `completionFactor`
  - `progress >= 0.9` 时增加权重
- `sourceFactor`
  - `favorite` 高于普通 `watch`
- `recencyFactor`
  - 越近越高

第一阶段建议先固定权重：

- 普通观看：`1.0`
- 完播加权：`1.2`
- 收藏加权：`1.5`

### 9.3 事件粒度建议

第一阶段推荐使用：

- 一次归一化行为
- 一主题一条贡献事件

不要在第一阶段尝试把“单次拖动进度条”“暂停恢复”等碎行为纳入兴趣事实层。

### 9.4 是否保留 `videoId`

建议：

- 保留 `videoId?: ID`
- 但只作为调试和可解释性字段

不要让兴趣统计逻辑依赖 `videoId`。

## 10. 兴趣快照计算

### 10.1 窗口

第一阶段仅支持：

- `7d`
- `30d`

### 10.2 统计来源

直接从 `INTEREST_CONTRIBUTION_EVENTS` 聚合，不再依赖：

- 扫全量 `WatchEvent`
- 扫全量 `Video`
- 每次现算视频归类

### 10.3 快照字段解释

- `watchDurationScore`
  - 窗口内该 topic 对应的观看时长贡献总和
- `eventCountScore`
  - 该 topic 贡献事件数量
- `favoriteScore`
  - 收藏类贡献值汇总
- `finalScore`
  - 综合排序值
- `ratio`
  - 在当前窗口内的占比

### 10.4 趋势分析

趋势建议在实现层动态计算：

- 当前 `7d` 对比前一段 `7d`
- 当前 `30d` 对比前一段 `30d`

第一阶段规则即可：

- 超过阈值为上升
- 低于阈值为下降
- 否则平稳

## 11. 运行链路规划

### 11.1 增量链路

触发点：

- `processVideoData`
- `processWatchEventData`
- `processFavoriteEventData`

增量链路职责：

1. 确保当前视频 tag 能映射到兴趣主题
2. 根据行为生成 `InterestContributionEvent`
3. 标记兴趣快照待刷新，或异步刷新最近窗口

这里要特别注意：

- 增量链路可以生成贡献事件
- 但不应在采集主链路中触发重型 LLM 或全量重算

### 11.2 回填链路

适用场景：

- 新增功能后的历史数据补算
- 调整映射规则后的批量重建
- 重新生成全部快照

回填内容：

1. 扫描已有标签，补齐 `TagInterestMapping`
2. 扫描历史观看/收藏数据
3. 为每个历史行为生成 `InterestContributionEvent`
4. 重算 `7d / 30d` 快照

推荐入口：

- 先放 `database/implementations`
- 再在 `ui/test-tools` 或独立调试页挂触发入口

## 13. 与 DataProcessor 的集成建议

`src/content/processors/data-processor.ts` 仍然是最适合接入兴趣贡献事件生成的入口，但要注意边界。

推荐做法：

### 13.1 `processVideoData`

职责：

- 不直接生成兴趣事件
- 仅确保标签存在
- 必要时补齐 `TagInterestMapping`

### 13.2 `processWatchEventData`

职责：

- 根据当前视频的 tag 映射结果
- 将观看行为归一化成一个或多个 `InterestContributionEvent`

注意：

- 这里生成的是兴趣分析专用事实
- 不要在这里做全量窗口重算

### 13.3 `processFavoriteEventData`

职责：

- 收藏时补记一条 `sourceType = favorite` 的贡献事件
- 作为强正反馈

### 13.4 不推荐的做法

- 在 `DataProcessor` 中直接做完整兴趣画像计算
- 在每次采集时全量重算所有快照
- 在采集时同步调用 LLM

### 14.1 当前可复用入口

- `src/engine/llm-client.ts`

后续应在其之上封装兴趣语义调用，不应复制新的 HTTP 客户端。

### 14.2 推荐包装层

建议新增：

- `src/engine/interest/interest-llm.ts`

- 校验返回 JSON
- 约束输出主题必须来自固定集合

### 14.3 使用策略

LLM 只在以下情况触发：

- 规则映射缺失
- 少量复杂 tag 需要判定

必须缓存：

- 同一个 tag 只归类一次

第一阶段不建议用 LLM 直接生成 `InterestContributionEvent`。

## 16. 验收标准

### 16.1 数据层

- 可初始化兴趣分析相关 store
- 可写入和读取 `InterestTopic`
- 可写入和读取 `TagInterestMapping`
- 可写入和读取 `InterestContributionEvent`
- 可写入和读取 `InterestSnapshot`

### 16.2 归一化层

- 已知 tag 能命中固定兴趣主题
- 一次观看可拆成多个兴趣贡献事件
- 无法识别的 tag 稳定落到 `其他`

### 16.3 统计层

- 能计算近 7 天兴趣 Top N
- 能计算近 30 天兴趣 Top N
- 收藏行为会明显提升相关主题分数
- 完播行为会影响贡献分值

### 16.4 UI 层

- 页面可展示兴趣列表
- 支持切换 `7d / 30d`
- 无数据时正常展示空状态
- LLM 未配置时功能仍能运行，只是回退到规则映射

## 17. 风险与演进方向

### 17.1 当前主要风险

目前没有一套天然的“逐次兴趣原子事件”数据源，第一阶段仍然需要从现有观看/收藏行为中归一化生成贡献事件。

这意味着：

- 历史回填逻辑会比较关键
- 初期映射规则质量会直接影响兴趣画像质量

### 17.2 第二阶段可扩展方向

- 将搜索行为也转成兴趣贡献事件
- 加入点赞、投币、评论行为
- 支持长短期兴趣融合模型
- 支持主题趋势折线图
- 支持主题下创作者偏好排名
- 支持兴趣驱动的推荐候选生成

---

## 19. 最终结论

当前项目最适合的兴趣分析第一版，不应该围绕 `WatchEvent` 或 `Category` 扩展，而应该建立一套独立的“兴趣主题 + tag 映射 + 兴趣贡献事件 + 兴趣快照”体系。

最推荐的实施顺序是：

1. 新建兴趣分析专用长期类型
2. 新建兴趣分析专用 store
3. 实现 `TagInterestMapping`
4. 实现 `InterestContributionEvent`
5. 实现 `InterestSnapshot`
6. 最后接 UI

这个方案的核心优点是：

- 语义边界清晰
- 统计链路简单
- 与当前系统低耦合
- 后续扩展推荐、趋势、搜索都更自然
