# 网页融合核心模块规范

本文档用于规划网页融合方向的第一批核心模块，聚焦以下五项能力：

- 推荐流过滤
- 悬浮信息卡片
- 卡片状态标记
- 视频页极简模式
- 预设模式切换

本文档目标不是描述具体代码实现，而是定义：

- 这些模块各自要解决什么问题
- 它们上线前系统需要具备哪些前置能力
- 当前项目已经具备哪些基础
- 模块之间如何协同
- 它们应当如何与现有分层保持平行
- 随着规模扩大，数据层需要如何演进

---

## 1. 文档定位

这五个模块并不是孤立功能，而是“插件与 B 站网页融合”的第一阶段主线。

它们的共同特点是：

- 都直接作用于用户浏览当下
- 都主要发生在前端网页层
- 都可以先以平行子系统方式接入当前架构
- 都会在后续逐步与数据层形成更深耦合

因此，本规范采用的核心设计思想是：

> 前期以前端增强系统为主，数据层轻支撑；后期逐步沉淀为依赖行为数据、内容特征和个性化规则的数据驱动系统。

---

## 2. 总体目标

第一阶段网页融合系统的总体目标是：

> 在不破坏当前系统主结构的前提下，让插件在用户浏览 B 站首页、推荐流和视频页时，成为一个实时可见、可控、可解释的增强层。

这个目标可以拆成五个直接结果：

1. 用户能控制推荐流里什么内容出现
2. 用户能在点击前更快判断视频是否值得看
3. 用户能识别哪些内容已经看过、重复出现或不值得投入
4. 用户能在视频页进入更专注的消费环境
5. 用户能一键切换不同浏览目标下的页面工作模式

---

## 3. 规划范围

本规范只覆盖网页融合第一阶段的核心模块，不覆盖以下内容：

- 完整推荐算法重写
- 深度个性化排序引擎
- 复杂系列视频识别系统
- 高自由度 CSS 生态
- 独立的复杂行为提醒系统

这些能力虽然与本阶段相关，但不属于当前规范的核心交付范围。

---

## 4. 当前系统现状评估

在开始定义模块前，需要先明确当前系统已经具备什么，以及缺什么。

### 4.1 当前系统已经具备的基础

从现有代码结构看，项目已经具备以下关键前置能力。

#### 4.1.1 清晰的前端与数据分层

当前项目已经明确强调：

- UI 层负责渲染和交互
- 查询层负责数据访问
- 数据库实现层负责持久化

这意味着网页融合模块可以继续遵守这一边界，不必破坏既有结构。

#### 4.1.2 Content Script 三层结构已经存在

当前 `src/content` 已形成：

- trigger 层
- collector 层
- forwarder 层

这为网页融合模块提供了非常好的参考。

虽然这套结构目前更偏“采集系统”，但同样说明：

- 页面事件监听可以被模块化
- DOM 提取逻辑可以被模块化
- 页面侧逻辑可以按职责拆分

后续网页融合模块也应该沿用类似思想，建立自己的：

- 页面检测
- DOM 解析
- UI 注入
- 页面状态同步

#### 4.1.3 已有视频页数据采集能力

当前系统已经能从视频页稳定收集：

- `Video`
- `WatchEvent`
- 收藏事件
- 关注事件
- 点赞 / 分享 / 投币事件

并且 `VideoDataCollector` 已经具备提取以下信息的能力：

- 标题
- 简介
- 标签
- UP 信息
- 发布时间
- 封面

这对以下模块非常关键：

- 悬浮信息卡片
- 卡片状态标记
- 视频页极简模式

因为这些模块都需要“认识当前视频或卡片对象”。

#### 4.1.4 已有设置存储能力

当前项目已存在基于 `APP_META` 的设置存储机制，且 `options` 页面已能读写统一设置。

这意味着网页融合模块前期不需要单独发明配置系统，可以先复用现有设置能力，逐步扩展为专门的网页融合配置。

#### 4.1.5 已有主题系统

当前主题系统已经支持：

- CSS 变量管理
- 主题切换
- 持久化

这意味着：

- 视频页极简模式的视觉形态有基础承接
- 信息卡片、状态标签、过滤标记可以沿用统一视觉语义
- 预设模式未来也可以和样式层协同

### 4.2 当前系统尚未具备的关键能力

虽然基础不错，但距离这五个模块真正落地仍有明显缺口。

#### 4.2.1 缺少“网页增强模块层”

当前项目有：

- 数据采集 content
- 弹出页 / 统计页 / 主题页 UI

但还没有一层专门负责“注入 B 站页面并增强网页交互”的模块体系。

这意味着需要新增一层平行于当前 content 采集链路的网页增强层。

#### 4.2.2 缺少推荐流卡片识别体系

当前采集重点在视频详情页和 UP 页。

但推荐流过滤、卡片状态标记、悬浮卡片都依赖：

- 首页卡片识别
- 搜索页卡片识别
- 分区页卡片识别
- 卡片数据提取

这部分目前还没有完整体系。

#### 4.2.3 缺少页面级模式状态系统

当前设置更偏全局静态设置。

但预设模式切换需要更细的状态维度：

- 当前启用哪个模式
- 模式作用于哪些页面
- 模式是否临时覆盖全局规则
- 页面打开时如何恢复状态

#### 4.2.4 缺少网页增强专用数据模型

前期许多功能可以靠前端临时状态运行，但规模扩大后会出现问题：

- 规则越来越多
- 页面元素越来越复杂
- 用户需要更细的控制
- 个性化判断需要历史数据支撑

这时就必须有专门的数据模型来承载网页融合系统。

## 5. 设计总原则

这批模块的设计必须遵守四条原则。

### 5.1 与当前系统平行接入

第一阶段不要把网页融合模块直接塞进现有采集主链路中。

更合适的做法是：

- 以独立前端增强子系统接入 content script
- 在配置和数据查询处与当前系统交互
- 在行为记录处与当前系统复用

这样做的好处是：

- 不破坏当前采集与统计系统稳定性
- 新模块可以快速试错
- 前端网页能力可以独立演进

### 5.2 轻数据依赖启动

第一阶段优先支持以下两类能力：

- 依赖公开页面信息即可工作的能力
- 依赖少量本地设置即可工作的能力

不要一开始就让所有模块依赖完整历史数据。

### 5.3 统一页面增强入口

这五个模块虽然功能不同，但都运行在同一网页中，因此需要共享：

- 页面上下文识别
- DOM 观察与重试机制
- 配置读取
- 注入与卸载生命周期

不能五个模块各自重复做一套页面监听。

### 5.4 可降级与可关闭

网页增强模块必须天然支持：

- 单模块关闭
- 模式级关闭
- 页面级降级
- DOM 识别失败后的容错退出

否则页面变动时会放大故障。

---

## 6. 模块总览

本阶段建议将网页融合第一批能力定义为一个独立系统：

### 系统名称

网页融合核心模块系统

### 子模块

1. 推荐流过滤模块
2. 悬浮信息卡片模块
3. 卡片状态标记模块
4. 视频页极简模式模块
5. 预设模式切换模块

### 公共基础模块

为了让上述模块稳定协作，建议额外规划四类公共基础模块：

1. 页面识别模块
2. 页面数据提取模块
3. 页面注入与生命周期模块
4. 网页融合配置模块

这四类模块不是最终功能，但它们是所有核心模块的前置基础。

## 7. 推荐流过滤模块规范

### 7.1 模块目标

推荐流过滤模块的目标是：

> 在首页、分区页、搜索页等卡片流场景中，基于规则决定哪些内容直接隐藏，哪些内容降权展示，哪些内容保留。

### 7.2 用户可见功能

建议第一阶段支持：

- 播放量阈值过滤
- 视频时长过滤
- 标题关键词过滤
- 标签包含 / 排除
- 来源过滤
- 直播 / 广告 / 番剧 / 电影等类型屏蔽
- 硬过滤与软过滤切换

### 7.3 依赖的前置能力

该模块至少需要系统具备：

- 推荐流页面识别能力
- 卡片节点识别能力
- 卡片基础元数据提取能力
- 卡片列表变化监听能力
- 页面规则读取能力

如果要支持更高级的过滤，还需要：

- 卡片详情补全能力
- 标签查询或 API 补拉能力
- 用户模式状态能力

### 7.4 当前系统已具备的部分

当前系统已经具备：

- 基础设置存储
- 视频标签和视频详情 API 能力
- 与视频相关的数据结构定义

但尚未具备：

- 推荐流卡片抽象
- 卡片页统一采集
- 页面级过滤执行器

### 7.5 模块输出

该模块最终应输出：

- 每张卡片的过滤结论
- 过滤原因
- 显示策略

推荐抽象为：

- `keep`
- `demote`
- `hide`

以及：

- `reasons[]`
- `matchedRules[]`

### 7.6 与其他模块关系

推荐流过滤是其他三个卡片类模块的重要前置：

- 悬浮信息卡片需要读取过滤原因
- 卡片状态标记需要展示过滤结果摘要
- 预设模式切换会影响过滤规则集

因此过滤模块不应只是“执行 DOM 隐藏”，而应先产出统一的卡片评估结果。

---

## 8. 悬浮信息卡片模块规范

### 8.1 模块目标

悬浮信息卡片模块的目标是：

> 在用户停留于视频卡片时，用一个轻量但高信息密度的浮层，帮助用户在点击前快速判断内容是否值得投入时间。

### 8.2 用户可见功能

建议第一阶段支持：

- 完整标题展示
- 简介摘要
- 标签展示
- UP 信息
- 时长与基础质量提示
- 当前模式匹配提示
- 过滤 / 标记原因展示

### 8.3 依赖的前置能力

该模块需要系统具备：

- 卡片 hover 触发机制
- 卡片对象识别
- 卡片元数据读取
- 卡片评估结果读取
- 浮层定位与渲染能力

如果需要展示更丰富信息，还需要：

- 异步详情补全能力
- 缓存层，避免频繁请求

### 8.4 当前系统已具备的部分

当前系统已经具备视频详情采集相关经验，尤其是：

- 标题
- 标签
- 简介
- UP 信息
- 发布时间

这说明字段理解层面基础已足够。

但当前缺少：

- 卡片 hover 生命周期管理
- 浮层渲染器
- 卡片数据缓存策略

### 8.5 模块输出

该模块输出的是一个悬浮 UI，而不是数据状态本身。

但为了可维护性，它应该依赖统一的输入模型，例如：

- `cardContext`
- `cardEvaluation`
- `videoInsight`

不要让浮层组件自己去解析 DOM 或自己去调用数据接口。

### 8.6 与其他模块关系

悬浮信息卡片是过滤模块和状态标记模块的解释层。

它应优先展示：

- 这张卡片为什么被保留 / 降权 / 隐藏
- 是否与当前模式匹配
- 是否已看过或重复刷到

因此它在交互上是“解释中枢”。

---

## 9. 卡片状态标记模块规范

### 9.1 模块目标

卡片状态标记模块的目标是：

> 用尽量轻量的视觉符号，让用户在扫视推荐流时快速识别内容状态，而不需要逐张点开判断。

### 9.2 用户可见功能

建议第一阶段支持：

- 已看过标记
- 重复刷到提示
- 高质量候选标记
- 疑似低价值提示
- 长视频但质量不足提示
- 当前模式推荐标记

### 9.3 依赖的前置能力

该模块需要：

- 卡片节点识别能力
- 卡片唯一标识提取能力
- 本地观看记录查询能力
- 卡片评估能力
- 轻量 UI 注入能力

### 9.4 当前系统已具备的部分

当前系统已经有：

- `WatchEvent`
- `Video`
- 收藏、点赞等行为事件

这意味着“已看过”“有过互动”“近期常见”这些状态是有机会逐步落地的。

但目前缺少：

- 针对网页卡片的实时查询接口
- 从 BV / 卡片上下文映射到本地数据的统一入口
- 面向页面增强的状态计算层

### 9.5 模块输出

该模块应输出卡片级状态集合，例如：

- `watched`
- `repeated`
- `high_quality`
- `low_value`
- `matched_mode`

每个状态还应带有：

- 视觉等级
- 文案
- 优先级

这样才方便多个状态并存时做合并渲染。

### 9.6 与其他模块关系

卡片状态标记与推荐流过滤关系最紧密。

两者区别是：

- 过滤负责决定“显示策略”
- 标记负责决定“视觉提醒”

因此状态标记不应直接决定隐藏逻辑。

它更适合消费统一评估结果，再以轻量方式表达。

---

## 10. 视频页极简模式模块规范

### 10.1 模块目标

视频页极简模式模块的目标是：

> 在视频详情页中为用户提供更专注、更低噪声的观看环境，减少页面其它区域对当前内容消费的打断。

### 10.2 用户可见功能

建议第一阶段支持：

- 隐藏评论区
- 隐藏相关推荐
- 隐藏部分互动数据
- 弱化无关信息区块
- 渐进隐藏与手动展开
- 一键进入专注模式

### 10.3 依赖的前置能力

该模块需要：

- 视频页识别能力
- 视频页 DOM 区块识别能力
- 样式注入能力
- 模式状态读取能力

如果要做得更稳定，还需要：

- 页面重新渲染后的自动重挂载能力
- 与主题系统协同的视觉变量

### 10.4 当前系统已具备的部分

当前系统已经：

- 非常熟悉视频页结构
- 能在视频页稳定运行 content script
- 有现成主题系统和 CSS 变量机制

因此五个模块里，视频页极简模式是最容易先落地的一项。

### 10.5 模块输出

该模块的本质输出不是数据，而是页面布局状态：

- 正常模式
- 轻极简模式
- 深度专注模式

并且每个模式应拆分为模块化隐藏策略，而不是一个巨大的样式文件。

### 10.6 与其他模块关系

视频页极简模式主要受预设模式切换控制。

它与推荐流过滤、卡片标记并非强耦合，但会共享：

- 配置系统
- 页面生命周期系统
- 样式系统

---

## 11. 预设模式切换模块规范

### 11.1 模块目标

预设模式切换模块的目标是：

> 将一组分散的过滤规则、标记规则和页面显示策略，打包成用户可以一键切换的使用场景模式。

### 11.2 用户可见功能

建议第一阶段支持：

- 高质量模式
- 长视频优先模式
- 学习模式
- 纯净模式

### 11.3 依赖的前置能力

该模块需要：

- 统一配置模型
- 规则组合能力
- 页面级状态广播能力
- 当前模式持久化能力

### 11.4 当前系统已具备的部分

当前系统已有：

- 设置持久化
- 主题切换经验

这意味着“模式切换”的基础设施可以从当前主题与设置机制中得到很多借鉴。

但还缺少：

- 面向网页融合功能的统一模式定义
- 模式变化后对子模块的统一通知机制

### 11.5 模块输出

该模块输出不直接是 UI，而是全系统共享的模式上下文，例如：

- 当前模式 ID
- 当前规则包
- 当前页面启用特性集

### 11.6 与其他模块关系

预设模式切换是五个模块中的控制中枢。

它直接影响：

- 推荐流过滤阈值
- 卡片状态标记策略
- 悬浮信息卡片展示侧重点
- 视频页极简模式启用强度

因此该模块虽然看上去像“设置功能”，但其实是系统级调度器。

## 12. 公共基础模块规划

为了让这五个模块稳定运行，建议在实现上先规划一组公共基础模块。

### 12.1 页面识别模块

职责：

- 识别当前是否为首页、搜索页、分区页、视频页、UP 页
- 提供统一页面类型枚举
- 对 SPA 场景中的 URL 变化进行响应

原因：

五个模块都需要知道自己当前处于哪类页面。

### 12.2 页面数据提取模块

职责：

- 抽象推荐卡片
- 提取卡片 BV、标题、UP、时长、标签入口、来源信息
- 抽象视频页区块节点

原因：

这是推荐流过滤、卡片状态标记和悬浮卡片的共同前置。

### 12.3 页面注入与生命周期模块

职责：

- 管理模块挂载与卸载
- 监听页面局部更新
- 管理 MutationObserver
- 统一重试与容错

原因：

B 站页面是高度动态的，如果没有统一生命周期层，五个模块都会各自重复监听页面变化。

### 12.4 网页融合配置模块

职责：

- 加载网页融合专用设置
- 提供模式解析结果
- 为各子模块下发配置快照
- 管理运行时配置更新广播

原因：

避免每个模块自己去读数据库或 settings。

---

## 13. 模块之间的交互关系

建议将这五个核心模块组织成如下关系：

```text
页面识别 / 生命周期
        ↓
页面数据提取
        ↓
预设模式切换 → 网页融合配置快照
        ↓
卡片评估层
   ├─ 推荐流过滤
   ├─ 卡片状态标记
   └─ 悬浮信息卡片

视频页模式控制
   └─ 视频页极简模式
```

这里最关键的是“卡片评估层”。

### 13.1 为什么要有卡片评估层

如果没有中间评估层，系统会变成：

- 过滤模块自己做规则判断
- 标记模块自己再做一遍判断
- 悬浮卡片模块自己第三次判断

这会带来：

- 重复逻辑
- 规则不一致
- 解释和实际行为不一致

因此更合理的设计是：

先统一得到“卡片评估结果”，再交给不同模块消费。

### 13.2 卡片评估层应输出什么

建议至少包括：

- 基础卡片信息
- 命中规则
- 过滤决策
- 标记状态
- 展示解释
- 模式匹配度

### 13.3 视频页极简模式为何独立

视频页极简模式与卡片类模块共享配置系统，但它不依赖卡片评估结果。

它更像页面布局策略模块，因此适合单独组织。

---

## 14. 建议的实现分层

考虑到当前系统分层清晰，网页融合模块建议采用如下平行结构：

```text
src/web-integration/
├── runtime/          # 页面识别、生命周期、上下文管理
├── extractors/       # 卡片 / 页面数据提取
├── evaluators/       # 规则评估、状态计算
├── overlays/         # 悬浮卡片、状态标记、提示UI
├── filters/          # 推荐流过滤执行器
├── focus-mode/       # 视频页极简模式
├── presets/          # 预设模式定义与解析
├── services/         # 配置读取、缓存、查询适配
└── types/            # 网页融合专用类型
```

### 14.1 与当前 content 层关系

建议关系如下：

- `src/content` 继续专注于数据采集
- `src/web-integration` 专注于页面增强

两者都运行于内容脚本环境，但职责不同。

### 14.2 与 UI 层关系

网页融合模块不是现有统计页 UI 的延伸，而是页面内 UI。

因此不建议把它放进 `src/ui/*`。

它更接近“页面运行时子系统”。

### 14.3 与数据库层关系

前期：

- 通过服务层读取设置
- 通过查询接口读取少量历史数据

后期：

- 逐步引入专门的数据模型
- 形成网页融合专用查询与统计能力

---

## 15. 前期如何与当前系统保持平行

这部分是本规范最关键的架构判断之一。

### 15.1 平行接入方式

第一阶段建议：

- 不改动当前采集主流程
- 不把网页融合逻辑塞进 `DataProcessor`
- 不让统计页直接承担网页增强职责

更合理的方式是：

1. content script 初始化后
2. 启动网页融合运行时
3. 网页融合运行时从设置和少量本地数据中读取上下文
4. 在页面内独立进行过滤、标记、浮层和极简模式注入

### 15.2 与当前数据层的最小交互

第一阶段建议只保留四类最小交互：

1. 读取全局设置
2. 读取当前模式
3. 查询视频是否已看过
4. 查询少量基础行为信息

这样可以让第一版尽快稳定。

### 15.3 为什么不要一开始深耦合

原因有三点：

1. 页面增强功能变化会非常频繁
2. DOM 兼容问题多，需要快速试错
3. 过早绑定数据层会放大改造成本

因此前期应优先保证：

- 模块边界清晰
- 页面行为稳定
- 配置和策略可快速迭代

---

## 16. 数据层当前可复用的能力

虽然网页融合系统前期以前端为主，但以下数据层能力可以直接复用。

### 16.1 设置持久化

当前 `APP_META` 已可用于：

- 模块开关
- 预设模式保存
- 页面极简模式状态
- 简单过滤规则保存

### 16.2 行为数据

当前已有行为数据可用于：

- 已看过标记
- 重复出现提示
- 简单兴趣推断辅助
- 后续模式推荐

主要包括：

- `WatchEvent`
- 收藏事件
- 互动事件

### 16.3 视频元数据

当前已有 `Video` 数据和视频 API 能力可用于：

- 卡片详情补全
- 标签识别
- 简单质量提示
- 视频页信息增强

## 17. 数据层需要新增的规划

虽然第一阶段可以轻依赖数据层，但如果希望这五个模块持续发展，数据层必须提前规划演进。

这里建议把数据层规划分成两个阶段。

### 17.1 第一阶段：轻支撑改造

目标：

让网页融合模块先可运行、可配置、可少量查询。

建议新增或扩展：

#### 17.1.1 网页融合设置模型

当前 `settings` 结构还比较偏基础应用配置。

建议未来增加独立配置分组，例如：

- 网页融合总开关
- 推荐流过滤配置
- 悬浮卡片配置
- 状态标记配置
- 视频页极简配置
- 当前预设模式

这部分前期可以先继续存入 `APP_META`，但结构上应独立命名，不要长期塞进同一层平面字段。

#### 17.1.2 轻量卡片状态查询接口

建议增加服务层能力，用于快速回答：

- 这个 BV 是否看过
- 最近是否重复出现
- 是否有收藏 / 点赞行为

重点不是新增大量存储，而是增加面向网页增强的查询入口。

#### 17.1.3 页面规则缓存

如果过滤规则和模式开始复杂化，需要：

- 当前规则包缓存
- 规则编译结果缓存
- 页面会话态缓存

这部分可以先在前端内存完成，但结构上要先规划接口。

### 17.2 第二阶段：深度数据驱动改造

目标：

让网页融合模块从静态规则系统升级为可学习、可解释、可个性化的系统。

建议规划以下数据模型。

#### 17.2.1 页面曝光事件

建议新增：

- 推荐卡片曝光记录
- 曝光位置
- 页面来源
- 曝光时间

作用：

- 判断“重复刷到”
- 分析推荐流质量
- 评估过滤效果

#### 17.2.2 页面反馈事件

建议新增：

- 用户手动屏蔽
- 用户手动降权
- 用户在卡片层点击了哪些快捷操作
- 用户切换了哪些模式

作用：

- 逐步形成用户侧推荐控制偏好
- 为后续排序和解释系统提供信号

#### 17.2.3 网页评估快照

建议新增面向卡片的评估结果缓存，例如：

- 某视频最近一次评估分
- 过滤原因
- 标记状态
- 模式适配结果

作用：

- 避免页面频繁重复计算
- 为悬浮信息卡片提供更快响应
- 为用户解释保留依据

#### 17.2.4 模式与规则实体化

当前预设模式可以先写死在前端。

但一旦模式需要支持：

- 用户自定义
- 多设备迁移
- 复杂规则组合

就必须把“模式”和“规则包”作为正式数据实体。

#### 17.2.5 页面布局偏好数据

视频页极简模式和页面模式切换发展到后期，会形成用户偏好数据，例如：

- 哪些区块常被隐藏
- 哪些模式常被启用
- 哪些页面需要更强干预

这些都值得沉淀为长期配置与统计基础。

## 18. 数据层建议的结构演进方向

从长期看，建议把网页融合能力也纳入数据层正式分区。

例如未来可以考虑增加一层：

### Web Integration Layer

候选对象包括：

- integration_settings
- card_exposure_events
- card_feedback_events
- card_evaluation_snapshots
- preset_profiles
- page_layout_preferences

这样做的好处是：

- 网页融合系统从“前端功能集合”升级为“有长期状态的数据系统”
- 后续推荐控制、行为认知、解释系统会更容易打通

## 19. 各模块的前置能力清单

为了便于排期，这里将五个模块的前置能力做一次统一汇总。

### 19.1 第一优先级前置能力

这些是五个模块普遍依赖、应最先补齐的。

1. 页面识别系统
2. 推荐卡片识别与抽象
3. 页面生命周期与重挂载机制
4. 网页融合配置读取
5. 当前模式上下文

### 19.2 第二优先级前置能力

这些能力会显著提升体验，但可以晚于第一批基础设施。

1. 卡片详情补全
2. 本地历史状态查询
3. 卡片统一评估层
4. 悬浮卡片渲染器
5. 轻量页面缓存

### 19.3 第三优先级前置能力

这些能力更偏中长期演进。

1. 页面曝光记录
2. 页面反馈记录
3. 模式实体化
4. 评估结果缓存
5. 个性化规则学习

## 20. 推荐的研发顺序

为了降低风险，建议按“基础设施优先、先易后难、先稳定后智能”的顺序推进。

### 第一阶段：公共底座

先做：

- 页面识别模块
- 生命周期模块
- 推荐卡片抽象
- 网页融合配置模块
- 预设模式定义

### 第二阶段：最容易形成用户感知的功能

优先做：

- 视频页极简模式
- 推荐流过滤
- 卡片状态标记

原因：

- 实现边界更清晰
- 风险可控
- 用户价值直观

### 第三阶段：解释能力

再做：

- 悬浮信息卡片

原因：

- 它需要前面的卡片评估结果和状态信息作为支撑

### 第四阶段：数据层深化

最后规划：

- 页面曝光与反馈
- 评估快照
- 模式实体化

---

## 21. 最终架构判断

这五个模块最适合被定义为：

> 一个与当前采集系统平行、以页面增强为中心、初期轻依赖数据层、长期逐步数据驱动化的网页运行时系统。

这意味着它们不应被拆散到：

- 统计页逻辑里
- 现有 DataProcessor 主链路里
- 零散的 content trigger 里

而应当拥有自己的边界、公共底座和演进路线。

---

## 22. 一句话结论

第一阶段最合理的做法，不是直接去做五个分散功能，而是先把它们规划成一个统一的网页融合子系统：

- 前端先平行接入
- 公共能力先打底
- 数据层先轻支撑
- 长期再逐步深耦合

这样既能快速落地用户最能感知的功能，也不会在后期扩展时被现有结构反噬。
