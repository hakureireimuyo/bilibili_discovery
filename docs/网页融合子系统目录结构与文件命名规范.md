# 网页融合子系统目录结构与文件命名规范

本文档用于规范网页融合子系统的目录结构、文件职责、命名方式和层间边界。

适用范围：

- `src/web-integration/` 下的全部模块
- 运行在 B 站页面内的网页增强逻辑
- 与推荐流过滤、悬浮信息卡片、卡片状态标记、视频页极简模式、预设模式切换相关的代码

本文档目标不是描述具体功能规划，而是解决工程落地问题：

- 代码应该放在哪
- 一个文件应该负责什么
- 哪些模块可以互相依赖
- 哪些依赖关系必须禁止
- 文件应该如何命名，避免越写越乱

---

## 1. 子系统定位

网页融合子系统是一个独立于当前 `src/ui` 与 `src/content` 采集链路的页面运行时系统。

它的主要职责是：

- 感知当前页面类型
- 识别推荐卡片和视频页区块
- 在页面中注入过滤、标记、悬浮卡片和极简模式
- 消费配置与少量本地数据
- 输出即时可见的网页增强效果

它不是：

- 统计页 UI 子模块
- 数据采集 trigger 的附属逻辑
- 数据库实现层的延伸

因此它必须有自己独立的目录结构和命名规范。

## 2. 总体目录结构

网页融合子系统建议统一放在：

```text
src/web-integration/
├── index.ts
├── README.md
├── runtime/
├── types/
├── presets/
├── extractors/
├── evaluators/
├── filters/
├── overlays/
├── focus-mode/
├── services/
└── styles/
```

各目录职责如下：

- `index.ts`
  - 子系统统一入口
- `README.md`
  - 子系统说明、入口关系、开发约束
- `runtime/`
  - 页面识别、导航监听、生命周期管理、模块挂载
- `types/`
  - 网页融合专用类型定义
- `presets/`
  - 预设模式定义与模式解析
- `extractors/`
  - 页面 DOM 提取逻辑
- `evaluators/`
  - 规则评估、状态计算、解释结果生成
- `filters/`
  - 推荐流过滤执行器
- `overlays/`
  - 页面浮层、卡片标签、悬浮卡片等 UI 注入逻辑
- `focus-mode/`
  - 视频页极简模式相关逻辑
- `services/`
  - 配置读取、缓存、历史状态查询等服务层
- `styles/`
  - 网页融合模块专用样式

## 3. 目录分层原则

### 3.1 以职责分层，不以功能堆文件

不要把所有与“推荐流过滤”相关的代码都塞进一个目录。

正确做法是按职责拆开：

- 抽取卡片：放 `extractors/`
- 评估卡片：放 `evaluators/`
- 执行隐藏：放 `filters/`
- 渲染标记：放 `overlays/`

这样有两个好处：

- 代码更容易复用
- 过滤、标记、悬浮解释可以共享同一份评估结果

### 3.2 页面运行时逻辑单独放在 runtime

任何与以下内容相关的逻辑，都优先放进 `runtime/`：

- 页面识别
- URL 切换监听
- 初始化时机
- 模块挂载与卸载
- `MutationObserver` 生命周期

不要把这类逻辑散落到业务模块里。

### 3.3 服务层负责“读配置与取数据”

任何需要与现有设置、查询、缓存、本地历史记录打交道的逻辑，都优先进入 `services/`。

不要让：

- `overlays/` 直接查数据库
- `filters/` 直接读 settings
- `focus-mode/` 直接访问 `APP_META`

## 5. 各目录职责规范

### 5.1 `runtime/`

职责：

- 初始化网页融合子系统
- 识别页面类型
- 监听页面切换
- 统一挂载和卸载子模块
- 管理全局运行时事件

可以做的：

- 创建模块实例
- 管理页面上下文
- 处理导航切换
- 统一清理监听器

不可以做的：

- 直接执行过滤规则
- 直接渲染悬浮信息卡片细节
- 直接读取数据库实现层

### 5.2 `types/`

职责：

- 定义网页融合专用类型

可以做的：

- 定义 interface / type / enum
- 定义轻量常量类型

不可以做的：

- 放业务逻辑
- 放 DOM 操作
- 放运行时代码

### 5.3 `presets/`

职责：

- 定义预设模式
- 解析模式规则
- 输出统一配置快照

可以做的：

- 定义模式常量
- 合并用户配置和模式规则

不可以做的：

- 直接操作 DOM
- 直接渲染页面元素

### 5.4 `extractors/`

职责：

- 从页面中提取卡片与区块信息
- 将不稳定 DOM 转换成稳定上下文对象

可以做的：

- 使用选择器读取页面元素
- 提取卡片字段
- 标准化原始文本和链接

不可以做的：

- 做规则判断
- 做过滤结论
- 注入悬浮 UI

### 5.5 `evaluators/`

职责：

- 对卡片或页面上下文进行业务评估
- 输出统一评估结果

可以做的：

- 组合规则
- 计算标记状态
- 生成解释文案

不可以做的：

- 直接执行 DOM 隐藏
- 直接渲染角标或浮层

### 5.6 `filters/`

职责：

- 根据评估结果对推荐流卡片执行显示控制

可以做的：

- 应用 `keep / demote / hide`
- 设置类名
- 控制布局与样式

不可以做的：

- 自己重新做一遍规则判断
- 自己去查询历史行为

### 5.7 `overlays/`

职责：

- 渲染页面覆盖层 UI
- 管理悬浮卡片和卡片状态标签

可以做的：

- 创建 DOM 节点
- 注入卡片标签
- 渲染 hover 浮层
- 管理定位和显示时机

不可以做的：

- 直接访问数据库
- 自己判断过滤规则

### 5.8 `focus-mode/`

职责：

- 管理视频页极简模式
- 应用区块隐藏和弱化策略

可以做的：

- 基于区块提取结果应用页面策略
- 控制展开和恢复

不可以做的：

- 负责推荐流卡片逻辑
- 负责模式存储本身

### 5.9 `services/`

职责：

- 提供配置、缓存、历史状态、调试等公共服务

可以做的：

- 读取网页融合设置
- 查询本地历史状态
- 使用已有 API 或查询接口补全数据
- 做页面会话缓存

不可以做的：

- 直接渲染 UI
- 直接持有复杂 DOM 状态

### 5.10 `styles/`

职责：

- 存放网页融合专用样式

可以做的：

- 定义类名对应视觉效果
- 使用主题变量

不可以做的：

- 夹杂业务逻辑说明
- 写与功能无关的大量通用页面样式

## 6. 文件命名规范

### 6.1 总体原则

统一使用：

- 小写字母
- 中划线连接
- 明确表达职责

例如：

- `page-detector.ts`
- `card-extractor.ts`
- `card-evaluator.ts`

不要使用：

- 含糊命名
- 大量缩写
- 功能与职责混杂的命名

例如以下命名不推荐：

- `utils2.ts`
- `page-handler.ts`
- `manager.ts`
- `card-all-in-one.ts`

### 6.2 类型文件命名

类型文件建议使用名词型命名。

示例：

- `page.ts`
- `settings.ts`
- `card-context.ts`
- `card-evaluation.ts`
- `focus-mode.ts`

如果目录中只有一个类型文件，也可以保留 `types.ts`，但网页融合子系统更推荐按领域拆分，不建议所有类型都放一个 `types.ts`。

### 6.3 提取器文件命名

统一使用：

- `[target]-extractor.ts`

示例：

- `card-extractor.ts`
- `video-page-extractor.ts`

辅助提取逻辑统一放：

- `extractor-helpers.ts`

### 6.4 评估器文件命名

统一使用：

- `[target]-evaluator.ts`
- `[target]-rules.ts`

示例：

- `card-evaluator.ts`
- `card-status-evaluator.ts`
- `filter-rules.ts`

### 6.5 服务文件命名

统一使用：

- `[domain]-service.ts`

示例：

- `integration-settings-service.ts`
- `card-data-service.ts`
- `card-history-service.ts`

缓存类服务统一命名为：

- `[domain]-cache.ts`
- `session-cache.ts`

### 6.6 运行时文件命名

运行时文件建议表达“监听器 / 运行时 / 事件 / 边界”职责。

示例：

- `integration-runtime.ts`
- `navigation-watcher.ts`
- `runtime-events.ts`
- `error-boundary.ts`

### 6.7 覆盖层 UI 文件命名

覆盖层文件名建议直接对应 UI 形态。

示例：

- `card-badges.ts`
- `info-hover-card.ts`
- `hover-controller.ts`
- `overlay-positioning.ts`

### 6.8 极简模式文件命名

极简模式文件统一使用：

- `focus-mode-*`
- `video-focus-mode.ts`

示例：

- `focus-mode-types.ts`
- `video-focus-mode.ts`
- `focus-mode-toggle.ts`

## 7. 导出文件规范

### 7.1 每个子目录建议提供 `index.ts`

例如：

```text
src/web-integration/evaluators/
├── index.ts
├── card-evaluator.ts
├── card-status-evaluator.ts
└── filter-rules.ts
```

这样做的好处是：

- 外部引用路径更稳定
- 子目录内部可重构而不影响上层过多 import

### 7.2 根目录 `index.ts` 只暴露公共入口

`src/web-integration/index.ts` 应优先暴露：

- 子系统启动入口
- 子系统关闭入口
- 少量公共类型

不要把所有内部模块都从根目录一股脑导出。

### 7.3 内部模块优先从子目录 index 导入

推荐：

```ts
import { evaluateCard } from "../evaluators/index.js";
```

不推荐：

```ts
import { evaluateCard } from "../evaluators/card-evaluator.js";
```

只有在目录内部实现细节需要精细拆分时，才允许直接引用具体文件。

## 8. 层间依赖规范

这是最重要的规范之一。

### 8.1 允许的依赖方向

推荐依赖方向如下：

```text
runtime
  ↓
presets / services / extractors
  ↓
evaluators
  ↓
filters / overlays / focus-mode
```

`types/` 可以被所有目录依赖。

### 8.2 明确禁止的依赖

以下依赖必须禁止：

- `overlays/` 直接依赖数据库实现层
- `filters/` 直接依赖 `settings-repository.impl.ts`
- `extractors/` 直接依赖 `evaluators/`
- `runtime/` 直接内嵌大量业务规则
- `focus-mode/` 直接复写推荐流逻辑

### 8.3 与现有系统的依赖边界

网页融合子系统允许依赖：

- `src/content` 的少量公共能力
- `src/api` 的公开视频详情接口
- `src/themes` 的主题变量与主题管理
- `src/database` 的高层服务或稳定实现入口

但不建议依赖：

- `src/ui/*` 页面模块内部实现
- 统计页局部工具函数
- 与网页增强无关的页面专属 DOM 代码

## 9. 文件职责边界示例

### 9.1 `card-extractor.ts` 应该做什么

应该做：

- 找到卡片 DOM
- 提取卡片标题、BV、时长、UP
- 返回 `CardContext`

不应该做：

- 判断它是不是低质量
- 直接隐藏卡片
- 直接注入 hover 浮层

### 9.2 `card-evaluator.ts` 应该做什么

应该做：

- 输入卡片上下文
- 输入当前模式快照
- 输出 `CardEvaluation`

不应该做：

- 改 DOM
- 发请求渲染 UI

### 9.3 `feed-filter.ts` 应该做什么

应该做：

- 根据 `CardEvaluation` 给卡片加类名
- 根据策略隐藏或降权

不应该做：

- 再次解析标题关键词
- 自己决定规则阈值

### 9.4 `info-hover-card.ts` 应该做什么

应该做：

- 接收已经准备好的展示数据
- 渲染 hover 卡片 UI

不应该做：

- 自己去读数据库
- 自己去猜测是否已看过

## 10. 样式文件规范

### 10.1 按能力拆样式文件

建议：

- `feed-filter.css`
- `overlays.css`
- `focus-mode.css`

不建议：

- 一个超大 `web-integration.css`

### 10.2 样式类名前缀统一

建议统一前缀：

- `bdi-`

示例：

- `bdi-card-hidden`
- `bdi-card-demoted`
- `bdi-hover-card`
- `bdi-focus-mode-light`

这样可以避免与 B 站原页面类名冲突。

### 10.3 优先使用主题变量

样式中优先使用：

- `--theme-*`

不要硬编码过多颜色值。

## 11. 公共工具文件规范

### 11.1 helper 文件只做轻工具

例如：

- `extractor-helpers.ts`
- `filter-helpers.ts`
- `focus-mode-helpers.ts`

这些文件只应包含：

- 文本标准化
- DOM 判断辅助
- 小型纯函数

不要把核心业务逻辑都塞进 helper。

### 11.2 不允许出现“万能 utils”

不推荐新增：

- `utils.ts`
- `common.ts`
- `misc.ts`

原因：

- 容易变成无边界堆放区
- 后续难以维护

更合理的方式是按领域命名：

- `extractor-helpers.ts`
- `overlay-helpers.ts`
- `runtime-events.ts`

## 12. README 与文档要求

### 12.1 根目录必须有 README

`src/web-integration/README.md` 应至少说明：

- 子系统定位
- 与 `src/content` 的关系
- 各子目录职责
- 初始化入口

### 12.2 每个复杂子目录建议有局部 README

对于以下目录，建议后续补充局部文档：

- `runtime/`
- `evaluators/`
- `overlays/`
- `focus-mode/`

### 12.3 文档要优先写边界

README 最重要的不是“这里有什么文件”，而是：

- 这个目录能做什么
- 不能做什么
- 应该依赖谁

## 14. 禁止事项清单

为了避免目录结构快速失控，以下行为建议明确禁止：

1. 不要把网页融合逻辑直接写回 `src/content/index.ts`
2. 不要把悬浮信息卡片写进 `src/ui/*`
3. 不要让 `overlays/` 直接访问数据库实现层
4. 不要让 `filters/` 和 `overlays/` 各自维护一套规则判断
5. 不要在 `extractors/` 中混入页面显示策略
6. 不要建立无边界的 `utils.ts`
7. 不要把所有网页融合类型都堆进一个巨大 `types.ts`

---

## 15. 一句话结论

网页融合子系统的目录设计核心不是“按功能堆文件”，而是：

- 运行时单独管生命周期
- 提取器只做页面抽象
- 评估器只做规则判断
- 过滤器和覆盖层只做执行与展示
- 服务层统一承接配置和数据读取

只要坚持这条线，后续模块再增多，目录也不会很快失控。
