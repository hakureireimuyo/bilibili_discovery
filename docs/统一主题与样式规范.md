# 统一主题与样式规范

## 目标

这份文档用于指导项目从“每个页面单独 CSS”向“统一主题体系 + 可复用样式层”迁移。它面向后续重构和新增页面，帮助保证：

- 项目风格统一
- 主题控制深度足够
- 页面样式可维护、可复用
- 组件样式与主题解耦


## 现状分析

当前项目的 CSS 设计特点：

- 每个页面/组件拥有独立 CSS 文档
- 主题模块存在，但只控制配色、边框、阴影等部分变量
- 主题与页面样式耦合度仍然偏高
- 本地样式覆盖较多，难以统一视觉体验

这导致的问题是：

- 统一体验难实现
- 主题只能覆盖“表面颜色”，不能控制整体风格
- 重复样式多，维护成本高
- 新页面开发时容易产生新的“孤立风格”


## 设计原则

### 1. 主题控制“语义变量”而非“具体颜色值”

主题应提供一套语义化变量，例如：

- `--theme-page-bg`
- `--theme-surface-base`
- `--theme-border-primary`
- `--theme-text-primary`
- `--theme-shadow-medium`
- `--theme-control-radius`

页面样式只消费这些变量，不直接写死 `#fff`、`rgba(...)`、`px` 等颜色值。

### 2. 公共样式层必须优先于页面样式层

建立一套基础样式库，包含：

- 基础布局类
- 通用组件基类
- 文本/标题/列表样式
- 常用控件样式
- 响应式/间距律

页面 CSS 只需要关注“页面特性”和“局部差异”，而不是重复定义基础外观。

### 3. 组件样式要可复用、可组合

避免每个页面自己写一套按钮、卡片、标签样式。尽量设计：

- `.btn`, `.btn--primary`, `.btn--secondary`
- `.card`, `.card--compact`
- `.badge`, `.badge--success`
- `.panel`, `.panel--bordered`

当页面需要特殊样式时，使用语义类或按需扩展，而不是重新定义整体外观。

### 4. 保持命名规范一致

推荐统一命名规则：

- BEM 或语义类
- 主题变量使用 `--theme-*`
- 组件变量使用 `--component-*`
- 通用 CSS 类使用 `u-` / `is-` 等前缀


## 目录结构建议

在 `src/ui` 下新增或改造以下文件结构：

```text
src/ui/shared/
  base.css
  theme-vars.css
  utils.css
  components.css
```

说明：

- `base.css`：全局重置、排版、字体、默认布局
- `theme-vars.css`：主题变量声明与 `:root` 默认值
- `utils.css`：通用工具类、间距类、文本类
- `components.css`：通用按钮、卡片、输入框、弹窗等基础组件样式

页面目录只保留自己的局部样式：

```text
src/ui/popup/popup.css
src/ui/options/options.css
src/ui/stats/stats.css
```

页面 CSS 仅用于该页面特有结构、特定布局和场景样式。


## 主题体系升级方案

### 1. 扩展主题变量

当前主题配置已经有 `colors`、`border`、`shadow`、`tagColors`、`heatmapColors`。

我们建议在现有基础上补充更多“语义层次”变量：

- page
  - `--theme-page-bg`
  - `--theme-page-gradient-start`
  - `--theme-page-gradient-end`
- surface
  - `--theme-surface-primary`
  - `--theme-surface-muted`
  - `--theme-surface-overlay`
- text
  - `--theme-text-primary`
  - `--theme-text-secondary`
  - `--theme-text-tertiary`
  - `--theme-text-inverse`
- border
  - `--theme-border-primary`
  - `--theme-border-secondary`
- control
  - `--theme-control-radius`
  - `--theme-control-gap`
- shadow
  - `--theme-shadow-light`
  - `--theme-shadow-medium`
  - `--theme-shadow-dark`
- accent
  - `--theme-primary`
  - `--theme-secondary`
  - `--theme-accent`
  - `--theme-success`
  - `--theme-warning`
  - `--theme-danger`
  - `--theme-info`

这些变量既能控制颜色，也能控制空间感、质感、层次感。

### 2. 让主题变量成为“全局入口”

`src/themes/theme-variables.ts` 应该输出完整的变量映射，并由主题入口注入到 `document.documentElement`。

页面 CSS 不必再写死“背景色/边框色”等具体语义，只要使用：

```css
background: var(--theme-surface-primary);
color: var(--theme-text-primary);
border: 1px solid var(--theme-border-primary);
box-shadow: 0 8px 24px var(--theme-shadow-medium);
```

### 3. 将主题扩展到非色彩风格

主题不仅控制色彩，还应控制：

- 圆角：`--theme-control-radius`
- 间距节奏：`--theme-control-gap`
- 字体尺寸层次（必要时可提供样式规范）
- 玻璃/磨砂/表面样式的深浅

这样就能让主题模块真正“深度影响界面”，而不是只影响“配色”。


## 具体迁移步骤

### 步骤 1：建立基础样式库

在 `src/ui/shared/` 里创建以下文件：

- `base.css`
- `theme-vars.css`
- `utils.css`
- `components.css`

并在页面入口统一引入：

```ts
import '../../shared/theme-vars.css';
import '../../shared/base.css';
import '../../shared/utils.css';
import '../../shared/components.css';
```

### 步骤 2：提取公共变量与通用样式

从现有页面 CSS 中抽取：

- 颜色、背景、边框、阴影相关变量
- 卡片、面板、按钮、标签、公用布局的样式
- 通用状态类（`.is-loading`, `.is-error`, `.is-active`）

例如：

```css
.card {
  padding: 16px;
  border-radius: var(--theme-control-radius);
  background: var(--theme-surface-primary);
  border: 1px solid var(--theme-border-primary);
  box-shadow: 0 8px 24px var(--theme-shadow-light);
}

.btn {
  display: inline-flex;
  align-items: center;
  justify-content: center;
  padding: 10px 16px;
  border-radius: calc(var(--theme-control-radius) * 1.2);
  background: var(--theme-primary);
  color: var(--theme-text-inverse);
}
```

页面样式只保留：

- 当前页面独有的布局结构
- 特殊状态或交互
- 不可复用的视觉效果

### 步骤 3：重构页面 CSS，改为“补差”模式

将原本完整的页面样式改为：

- 复用 `shared` 中的通用类
- 使用 `theme` 变量替代硬编码颜色
- 用 `utils` 类替代简单布局需求
- 仅保留页面特有结构样式

例如 `popup.css` 中：

- 把 `.panel`, `.hero`, `.health-badge` 等通用视觉抽成 `shared/components.css`
- 留下 `.focus-grid`, `.hero-copy` 这类页面特定结构

### 步骤 4：强化主题适配测试

每次新增主题变量后，必须：

- 确认浅色/深色切换可用
- 验证当前页面在不同主题下无严重对比问题
- 检查 border/text/background 变量是否正确映射
- 重点控制“暗色模式的背景与文本层次”

建议使用单页调试或快速页面切换脚本验证。

### 步骤 5：持续补齐“语义变量”

当发现某个页面仍然写死颜色或样式时，先判断是否属于通用语义：

- 是通用的样式：应新增主题变量并提取到 `shared`。
- 只是当前页面特有的视觉差异：继续保留在页面 CSS，但仍然使用主题变量。

例如 `popup` 的 `hero` 渐变背景，可以保留结构，但要尽量使用：

- `--theme-primary`
- `--theme-secondary`
- `--theme-surface-primary`

而不是直接写死 `#8b9dc3`。


## 推荐实践

### 语义变量优先

页面层不要直接消费 `colors.primary`、`colors.secondary`，而是消费“语义词”：

- `--theme-surface-primary`
- `--theme-border-primary`
- `--theme-text-secondary`
- `--theme-shadow-medium`

这样当主题配色体系变化时，页面无需改动。

### 组件类语义化

避免直接使用 `.btn-blue`、`.card-white` 等具象类名。

推荐：

- `.btn`, `.btn--primary`, `.btn--secondary`
- `.badge`, `.badge--success`, `.badge--warning`
- `.panel`, `.panel--dense`

### 单一职责页面 CSS

页面 CSS 只做“页面级布局和特定风格补差”。

如果样式可以放入 `shared/components.css`，就应当放入。

### 主题变量不要过细

不鼓励把每一个具体按钮颜色都写成变量。优先保持语义层级：

- 通用主题变量
- 通用组件样式
- 页面特有增强样式

只有在“不同主题需要差异化渲染”的场景，才新增变量。


## 实施建议

### 先从 `popup` 页面开始重构

`popup.css` 目前已有大量可提取的通用视觉样式，适合作为第一批重构对象。

目标：

- 抽出 `base.css` / `theme-vars.css` / `components.css`
- 保留 `popup.css` 的特定布局和文字样式
- 将 `popup` 的配色全部改为 `var(--theme-*)`
- 验证主题切换是否正常

### 再对 `options`、`stats`、`workbench` 进行批量提取

逐页迁移思路相同：

1. 找到重复外观样式
2. 抽成 `shared` 组件类
3. 替换页面 CSS
4. 保持页面结构样式独立

### 最终目标

- `shared` 样式库承担大部分视觉基础
- `theme` 体系控制颜色、间距、圆角、阴影等风格要素
- 页面 CSS 只做局部差异扩展
- 新增页面无需从零开始定义视觉风格


## 结论

这个方案的核心是：

- 搭建“主题 + 共享样式”双层架构
- 用 `CSS 变量 + 语义类` 统一风格入口
- 用 `页面补差` 替代 `页面自定义` 的写法

这样既保留了单页可定制能力，又能保证项目整体风格一致，主题模块也能真正“深度生效”。
